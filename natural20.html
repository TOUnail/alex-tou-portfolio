This app was my first attempt at a full-stack application. A social media app that uses the Dungeons and Dragons idea where a 20-sided die determines whether or not a user's action passes or fails.

The idea behind it stems from a game of Dungeons and Dragons while I'm consuming social media while I waited for my turn.

How does it work?

When a user submits a post, 2 random numbers ranging from 1 to 20 gets generated.

  • The first number is the user's number
  • The second number is a number the user has to "roll" to succeed.

If the user's number is greater than or equal to the second number, the user succeeds and a PASS badge will be displayed on the post. However, if the user's number is less than the second number, the user fails and a MISS badge will be displayed instead.

There is an exception when the user rolls a 1 or a 20. When a user rolls a 1, this is the lowest possible number that can be rolled. In Dungeons and Dragons, this is called a 'critical miss' where the user fails so badly, there are worse consequences than normal. Vice versa, when a user rolls a 20, this is the highest number that can be rolled. This is called a 'crit' or 'critical' where the user overwhelmingly succeeded that there are positive side effects that follow.

In Dungeons and Dragons, when the dice result is determined, the Dungeon Master will narrate how the user passes or fails. This is where the comment section comes in.

Other users will narate how you pass or failed your post.

How was it built?

This app is built with a FERN-stack:

Firebase

Express

ReactJS

NodeJS

All of the data is being stored in Firebase while an API provided by Express and NodeJS is being fetched and rendered on ReactJS. After dipping my toes into serverless functions and database management, I have a whole new respect for those who work in the back-end. Researching how to manage data efficiently took up a quarter of development time with a ton of trail and error.

Challenges

Originally, I was using Redux to manage data on the client-side. As a challenge, I decided to see if this could be done using ReactJS's built-in Context API. This is how I learned that Context API is not a Redux Replacement. First and foremost, Context is not a state management tool. In fact, Context doesn't manage anything. This was one of the issues I ran into when trying to convert everything Redux was handling to Context. All of the management had to be done using useState/useReducer.

The biggest issue I ran into was useReducer. Since Context relies on passing the state down to a component and useReducer changes the state value, any component that is linked via Context will re-render.
